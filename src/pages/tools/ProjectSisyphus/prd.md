# 项目代号：Project Sisyphus (西西弗斯认知引擎)
**文档版本:** V 0.1 (MVP)
**产品定位:** 基于“合意困难”与“渐隐式支架”的生成式学习引擎。
*(MVP 阶段以“实用英语场景生成”为外壳，内核为普适性的知识重构系统)*

---

## 🎯 一、 战略愿景与目标人群

### 1. 核心护城河 (The Moat)
市面上的 AI 教育产品（如多邻国、各类背单词App、解答题App）本质上在做“知识的喂养”与“多巴胺反馈”。
本产品的核心壁垒在于**“认知摩擦力的精准控制”**：不提供现成答案，通过 LLM 动态拆解知识点，逼迫用户大脑进行“提取练习 (Active Recall)”，重塑神经元连接。

### 2. 目标人群演进路线
*   **【当前 MVP 阶段】：高度自驱的成年人（“0号测试体”即你自己）**。需要突破英语口语/书面表达瓶颈，能忍受初期极高的认知消耗，追求真实的“知识留存率”。
*   **【中期扩展】：功利导向的高认知人群**。考研党（政治/专业课主观题）、自学编程转码者、CPA/法考备考者。
*   **【远期终局】：具备自驱力的新一代 K12/终身学习者**。作为底层 OS，替代传统教辅。

---

## 🛠️ 二、 MVP 核心业务流 (User Flow)

MVP 摒弃一切非必要功能（如注册登录页、商城、排行榜），核心是跑通**“学习 - 提取 - 巩固”**的认知闭环。

1.  **入场 (The Hook):** 打开应用，系统直接抛出**【今日生存场景】**（如：在国外遭遇车祸如何向警察描述）。
2.  **搏斗 (The Struggle):** 用户尝试输出，触发 AI “渐隐式支架”循环（只给提示，不给答案，动态降级难度）。
3.  **突破 (The Breakthrough):** 用户最终拼凑出正确表达，AI 给予情绪价值与深度解析。
4.  **沉淀 (The Extraction):** 后台静默提取本次新知识与用户错误，存入用户的“知识突触库”。
5.  **回响 (The Echo):** 次日，系统算法调度该知识点，大模型生成**全新的场景**进行间隔测试。

## 💻 三、 MVP 功能模块细分 (开发 TODO 级)

### 模块 1：前端交互组件 (Dynamic UI Render)
*理念：前端必须是“哑终端”，所有交互形式由后端的 JSON 驱动。*
*   **P0: 对话流主界面:** 极简的 Chat 界面。
*   **P0: 动态组件解析器:** 根据后端传来的 `dialogue_type` 渲染不同组件：
    *   `chat`: 正常的文本输入框。
    *   `cloze`: 句子填空组件（例如：I \_\_\_ never \_\_\_ there. 用户只能填空）。
    *   `reorder`: 单词卡片拖拽排序组件。
*   **P1: “求助 / 放弃”按钮 (Help/Give up):** 点击后主动要求 AI 降级支架（从 Level 1 降到 Level 2）。
*   **P1: “我不服”按钮 (Challenge):** 当 AI 判定错误时，用户可点击并陈述理由，触发特定 Prompt 进行重新裁判。

### 模块 2：AI 大脑层 (Prompt & LLM Services)
*理念：将 LLM 拆分为不同职能的 Agent，各司其职，避免单一 Prompt 过长导致幻觉。*
*   **P0: 核心苏格拉底导师 (The Tutor Engine):** 负责主聊天流，严格执行“不给答案”、“动态脚手架”指令。
*   **P0: 知识提炼机 (The Memory Extractor):** 异步执行。对话结束后，总结提取 `KnowledgeNode` (知识点/词汇/错误模式) 返回 JSON。
*   **P0: 场景生成器 (The Scenario Generator):** 接收复习清单，生成必须使用特定知识点的新场景。
*   **P1: 裁判引擎 (The Arbiter):** 专门处理用户点击“我不服”时的逻辑自洽检查。

### 模块 3：业务逻辑与算法层 (Backend)
*   **P0: 上下文截断管理器:** 保证传给 Tutor Engine 的上下文仅包含当前 Scenario 的描述和最近 3 轮对话，外加用户的核心画像标签。
*   **P0: “死锁”突破机制 (Deadlock Breaker):** 状态机记录单个知识点交互轮数。`If round >= 3`，强行覆盖 LLM 指令，输出最终答案，并标记该知识点掌握度为低。
*   **P0: 间隔重复算法库:** 接入开源算法（如 `FSRS` 或 `SuperMemo-2`）。每日零点计算出今日必须复习的 `KnowledgeNode` ID 列表。

## 🏗️ 四、 面向未来的架构设计 (Future-Proofing Strategy)

为了保证目前的 MVP 能够平滑扩展到“万物皆可学”的认知引擎，底层的数据库和抽象逻辑必须具备前瞻性。

### 1. 内容抽象化 (Content-Agnostic Database Schema)
不要在表结构里写死 `vocabulary` 或 `grammar`，必须抽象为 **Node (节点)**。

```python
class KnowledgeNode:
    # MVP 阶段可能是 "word: deposit"
    # 未来扩展阶段可能是 "math: 导数定义" 或 "code: Python 装饰器"
    node_id: str 
    domain: str # 学科领域 (e.g., 'esl', 'calculus', 'python')
    concept_description: str # 知识点核心定义
    mastery_score: float # 掌握度
    ...
```

### 2. 用户自建学习流 (BYOC - Bring Your Own Content)
*   **现在的形态：** AI 自动生成英语场景。
*   **未来的形态：** 系统增加一个 **“知识解构入口”**。
    *   用户丢入一份《CPA会计准则PDF》或一段《高等数学讲解视频》。
    *   **架构预留：** 后台增加一个 `Content Parse Engine` (内容解析引擎)，它负责把复杂的外部资料，拆解成一个个 `KnowledgeNode`，并存入用户的待学习库。
    *   随后的逻辑**完全复用 MVP**：Tutor Engine 提取这些 Node，生成相关的业务场景（例如：扮演审计师查账），逼迫用户提取知识。

### 3. 多模态与物理世界交互 (Multimodal Extension)
*   **架构预留：** 前端的组件解析器必须支持图片和语音。
*   **未来场景：** 学习机修/医学。系统发出一张引擎故障/X光照片。系统问：“指出异常在哪里，并说明原因。”（这只需在现有的 JSON `component_data` 里增加 `image_url` 字段即可）。

### 4. 社交与市场 (Marketplace - The Final Moat)
*   一旦底层逻辑跑通，“如何设计好的提问/场景”将成为最有价值的资产。
*   **未来形态：** 用户/名师可以在平台上建立自己的“认知图谱”，分享他们定制的“特定领域生存场景库”。比如：“硅谷大厂面试拷问场景包”、“刑法张三奇葩案例包”。

## 📈 五、 成功指标 (Metrics that Matter)

在 MVP 阶段，我们**不看**停留时长（太痛苦会导致时长短），**不看**活跃度，我们只看：

1.  **主动思考耗时 (Time-to-Resolve):** 用户在输入框停留、删除、修改的时间。越长，说明认知摩擦越有效（只要不流失）。
2.  **脚手架降级率 (Scaffolding Downgrade Rate):** 统计完成一个节点所需的平均提示层级。如果层级随时间逐渐降低，说明用户能力在真实提升。
3.  **大模型指令服从率 (LLM Adherence):** 通过日志监控，大模型“不慎直接泄露答案”的发生比例（目标控制在 5% 以下）。
4.  **7日后真实提取成功率:** 在第7天的不同场景下，用户能否在不需要提示的情况下，一次性正确使用 Day 1 学到的知识。
