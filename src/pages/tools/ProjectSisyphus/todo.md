# 🚀 Project Sisyphus (西西弗斯认知引擎) - MVP 开发任务清单

## 📍 核心交互流说明

本项目的核心不是传统的“用户问，AI答”，而是一个**状态机驱动的动态脚手架系统**。
1. **初始化：** 用户进入，后端查库获取今日需复习的知识点（节点），交由【场景生成大模型】生成一个带冲突的生存场景发送给前端。
2. **核心循环：** 用户尝试输入 -> 后端【导师大模型】评估 -> **后端严格输出 JSON** -> 前端解析 JSON 动态渲染出 UI 组件（自由输入框 / 填空题 / 乱序重组）。
3. **死锁熔断：** 同一知识点，用户如果连续失败 3 次，系统触发熔断，直接给出正确答案结束当前循环。
4. **异步沉淀：** 对话结束后，后端【提取器大模型】在后台静默分析刚刚的对话，提取新的知识节点和错误模式，更新到用户的“认知图谱数据库”中。

## 🎨 前端开发任务 (Frontend & UI)

*设计原则：前端是“纯粹的执行者”（哑终端）。不要在前端写任何关于英语、对错判断的逻辑。前端只负责解析后端下发的 JSON，并渲染成对应的视觉组件。*

### 模块 1：基础布局与状态管理
- [ ] **F-1.1 构建极简对话流主界面**
  - **描述:** 类似聊天软件的界面布局（全屏高度，底部固定输入区，中间消息滚动区）。
  - **验收标准:** 视觉清爽，无多余干扰。支持消息自动滚动到底部。
- [ ] **F-1.2 建立全局状态管理器**
  - **描述:** 管理当前的 `session_id`、聊天历史列表、以及系统加载/思考状态 (`is_thinking`)。
  - **验收标准:** 当 `is_thinking` 为 true 时，输入框禁用，并在界面显示呼吸灯或“AI 正在思考脚手架”的动画。

### 模块 2：动态组件渲染器 (Dynamic Renderer)
- [ ] **F-2.1 开发 `<ChatBubble />` (常规气泡组件)**
  - **描述:** 渲染后端的 `ai_speech` 和 `emotional_support`。
  - **验收标准:** 情绪安抚文本和实际的引导文本在视觉上有所区分（如情绪支持用偏淡的颜色）。
- [ ] **F-2.2 开发 `<ClozeInput />` (完形填空组件)**
  - **描述:** 当后端 `interaction_type === 'cloze'` 时渲染。解析后端下发的带有下划线的句子结构，并在下划线处渲染出 `<input>` 框。
  - **[未来考量]:** 这种组件化设计意味着未来如果后端要求渲染数学公式填空，前端只需新增一个对应的组件即可，无需改动核心流。
  - **验收标准:** 用户只能在填空处输入，点击提交后，将填入的词拼接成完整句子发送给后端。
- [ ] **F-2.3 开发 `<ReorderCards />` (乱序拖拽组件)**
  - **描述:** 当后端 `interaction_type === 'reorder'` 时渲染。解析后端下发的单词数组，渲染为可拖拽的卡片。
  - **验收标准:** 用户可以通过拖拽/点击改变单词顺序，排序完成后点击提交，将顺序连接成字符串发给后端。

### 模块 3：交互外设与按钮
- [ ] **F-3.1 实现底部复合输入区**
  - **描述:** 包含主文本输入框、发送按钮、以及一个微小的 `[求助/降级]` 按钮。
  - **验收标准:** 用户点击求助时，发送特定的系统指令给后端（如：“用户请求降低脚手架难度”），无需用户打字。
- [ ] **F-3.2 实现 `[我不服]` 挑战按钮**
  - **描述:** 在每一个 AI 的反馈气泡旁边，提供一个极小的图标按钮。点击后弹出一个轻量输入框让用户输入理由。
  - **验收标准:** 收集用户理由并调用 `/challenge` 接口，用裁判结果覆盖当前的 UI 状态。
- [ ] **F-3.3 耗时隐性打点逻辑**
  - **描述:** 记录用户从看到 AI 回复，到最终点击“发送”之间的停留耗时（毫秒）。
  - **验收标准:** 每次调用 `/chat` 接口时，Payload 中必须携带准确的 `time_taken_ms` 字段。
